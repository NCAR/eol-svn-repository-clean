Purpose

This document contains instructions for converting Purdue Mobile High Resolution Radiosonde Data into the EOL Sounding Composite (ESC) format.

As of 6 January 2014, we are waiting to hear back from the source regarding several data issues. Scot L. has advised that there will be more changes required.

Directory Structure

ARM/

5mb/The directory where the extracted 5mb pressure sounding files are stored.
build/The auto-generated directory for the compiled Java classes.
dayfiles/The directory where the generated day files are stored. This is the final processed data that will be put on line.
docs/The directory containing documentation about the processing and the data.
final/The directory where the QC'ed files are placed once they have been created.
logs/The directory where the error/check files from the QC are placed.
output/The directory where the CLASS files are put after they are generated.
raw_data/The directory where the raw data is stored.
software/The directory where the software for the conversion is stored.
src/Unused directory - required for autoqc to run.

Processing the Data

Initial Processing Setup

  1. Copy the raw data into the raw_data directory. Check the assumptions made in the script regarding things such as file names and locations. The raw data consists of EDT and *.tsv (containing lat/lon info) data in two separate files per sounding.

    For MPEX:

    • The raw data files were in a tab-delimited ascii format with everything on a single line in the files. It was determined by Linda Cully that the files were created on a Mac. We were able to use "tr" (the Unix translate program) to make the files readable for processing.

      tr '\r' '\n' < research.Purdue_sonde.20130612223305.skewT.txt > research.Purdue_sonde.20130612223305.skewT.txt.trans

Converting the Data

This step runs Perl code and should work on basically any machine. However, to run the autoQC and any other processing/data prep done by the build.xml (Ant) file, you must run on any machine with the proper Ant and Java setup, such as tikal.

Converting Purdue Mobile High Resolution Radiosonde data into the ESC format consists of the following steps.

  1. Change to the software directory.

  2. Edit the Purdue Mobile_Radiosonde_Converter.pl script. This converter merges separate EDT and *.tsv text files into single records keyed on altitude.

    • For MPEX:

      • BEWARE: This data had a lot of problems, so this converter has several hard-coded fixes that may not be appropriate for other sounding datasets. And, as of 6 January 2014, we are waiting to hear back from the source regarding several data issues. Scot L. has advised that there will be more changes required.
      • The date, time and sonde ID are taken from the file name.
      • The lat, lon, and alt information are obtained from the surface data record (the first line of the raw data).
      • A separate script, AddMissingColums.pl, was run on research.Purdue_sonde.20130604011707.skewT.txt as a pre-processing step to insert missing RH columns at the end of the file.
      • For research.Purdue_sonde.20130524001306.skewT.txt three lines were manually changed to remove a digit and spaces before the data.
      • Inserted code that made sure that missing wind values would also have missing wind direction.
      • Added code to check for missing pressure and stopped processing at that point. (Several files had missing pressure records for several lines at the end of the file.)
      • For research.Purdue_sonde.20130518230503.skewT.txt, Scot L. had indicated that a section of code had bad wind values and asked that all wind values in that section be set to missing. The beginning and ending line of that section had to be identified by hard-coding in the time and wind speed.
      • Several files had sections were "extra" columns were created by the insertion of tabs in the raw data. This caused the lat/lon data to appear in an unexpected column, so code was added to identify these lines and look in the correct location for lat/lon info.
      • Added a fix for the wind direction. It was not being converted properly. Scot L. noticed that they are in radians rather than degrees.
      • Scot L. requested that I set the pressure values in one of the data files to "missing" so code was added to accomplish that.
      • Hard-coded fixes for files that Scot specified should have records cut off after certain times.
      • Hard-coded fixes for various files where temperatures should be set to missing at particular times as indicated by Scot.

  3. Run the Purdue Mobile_Radiosonde_Converter.pl script. This will convert data to the ESC format.

  4. Run the format checker using the ant command:

    ant check-format-ESC

  5. Check the log file (generated by the format checker) in the output directory to determine if any format issues need to be addressed before proceeding with AutoQC.

Auto QC

This step works best on tikal, but can be done on any machine with the proper Ant and Java setup.

These steps execute the automatic QC on the processed files in the output directory, generate log files in the logs directory, and place the QC'ed data files into the final directory, as specified by the build.xml file. The final part of the automated process is to run the format checker on the QC'ed files. To run the sounding Automatic Quality Control Processing:

  1. Stay in/Change to the top level processing directory where the Ant build.xml file is located.

  2. Check the build.xml file for the autoqc target for the following:

    1. A command to copy the properties file to the build directory. There is typically a copy of the autoqc properties file (e.g., "us_plains_autoqc.properties") in the software directory.

    2. The java command(s) will be executed on the expected data in the expected locations (i.e., ensure that the directory names in the build.xml file are correct and what you want.)

    3. You have updated the project name in the build.xml, if included in the build.xml file.

  3. Perform the auto QC by executing the ant command:

    ant autoqc

  4. After all issues have been resolved, notify the Scientific Staff (Scot) that the data is ready for Visual QC. Also inform him of any issues noted so that he will be aware of them.

Visual QC

Once the auto QC is completed, email the scientific staff (i.e., Scot Loehrer) so that they can perform a visual QC on the data and give final approval of the processed data set, as required. Note that the visual qc is not performed on all sounding datasets.

Finalization

After the visual QC is completed, the data set is ready to be finalized (i.e, prep'd for CODIAC).

  1. Update this "How To" document to indicate the latest project this conversion was executed for and include any notes about upgrades made specifically for that project, as required. A copy of this document is generally found in the /docs directory. Place a copy of the updated doc in /net/web/dmg/html/software/conversions/upper_air/Purdue Mobile directory. The s/w repository page points to this doc in that web location, so this must be updated. The s/w repository page is located here.

  2. Update the dataset description (or EMDAC) document for this conversion. Work with the scientific staff to create an EMDAC document. Generally, start with an existing "similar" document and simply update it for this project and dataset. Generally, a copy of the EMDAC doc can be found in the /docs area.

  3. The next processing step is to create day files that can/will be loaded into the EMDAC/CODIAC database, along with the EMDAC dataset description document. The dayfiles creation step will generate day files sorted by nominal release date followed by station location as required by CODIAC. It's very important that the output dayfiles be in this expected sort order so that CODIAC can access all the soundings within a single dayfile. Do the following steps to create the day files.

    1. Ensure that you are in the same (top) directory as the build.xml file. Create the day files using the ant command:

      ant dayfiles

      This will create a file for each day where at least one sounding exists in the database directory or whichever directory is specified as the output directory in the build.xml file. If this dataset is to be put online "as is", these ar the dayfiles to load into CODIAC.

5mb Extraction

If the converted soundings are also going to be placed into a 5mb composite, the soundings need to have the proper 5mb pressure level soundings generated. This is done through the following steps:

  1. Generate the 5mb pressure level data files using the ant command:

    ant 5mb-extract

    This will generate a 5 mb sounding file for each sounding in the final directory and place it in the 5mb. It will also run some final checks to make sure the format is correct. Beware that the file directories listed in the build.xml file are what you expect and want.

The generated 5mb data files will then be available when the composite is generated.
