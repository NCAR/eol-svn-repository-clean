Purpose

This document contains instructions for converting MISS upper air (upsonde) sounding data from the ISF EOL format into the EOL Sounding Composite (ESC) format.

Top

Network Information

  • The ID-Type is 99.
  • The Platform is 121, Rawinsonde, NCAR/ISS.

    For PLOWS_2008-2009:

    The platform was originally set to "298, Rawinsonde, GAUS" but was changed to the ISS platform after consulting with the scientific staff (Scot).

    Scot's reply regarding choice of platform: "GAUS (or really MGAUS for mobile GAUS) is the radiosonde. But ISS (or really MISS) is the overall system (profiler, surface met, radiosonde). We tend to use ISS for the platform when the overall system is used for a project and GAUS when just the radiosonde system is used. In this case the entire ISS system was used so I'd go for that."

Directory Structure

MISS/

5mb/The directory where the extracted 5mb pressure sounding files are stored.
build/The auto-generated directory for the compiled Java classes.
database/The directory where the final processed data that will be put on line is stored.
dayfiles/The directory where the generated day files are stored.
docs/The directory containing documentation about the processing and the data.
final/The directory where the QC'ed files are placed once they have been created.
logs/The directory where the error/check files from the QC are placed.
output/The directory where the CLASS files are put after they are generated.
raw_data/The directory where the raw data is stored.
software/The directory where the software for the conversion is stored.
src/Unused directory.

Processing the Data

Initial Processing Setup

  1. Copy the raw data into the raw_data directory.

Converting the Data

This step runs Perl code and should work on basically any machine. However, to run the autoQC and any other processing/data prep done by the build.xml (Ant) file, you must run on any machine with the proper Ant and Java setup. Such as merlot.

Converting ISF EOL formatted MISS upsonde data into the ESC format consists of the following steps.

  1. Change to the software directory.

  2. Edit the ISS_Converter.pl script.

  3. Change hard-coded values for project-specific items such as Project, Network, State, etc.

    For PLOWS 2008-2009:

    • BEWARE: This project converted MISS (mobile ISS) data. To use the converter for ISS data, change the hard-coded "NETWORK" and Header ID to "ISS". This needs to be changed in the build.xml file also.

    • In the build_default_station code, the state was set to "IA." After the conversion was completed, it was learned that soundings were taken in Iowa and Wisconsin. The Station CD file was hand-edited to reflect a state code of "99" since there was more than one state.

    • The platform was originally set to "298, Rawinsonde, GAUS" but was changed to the ISS platform "121 Rawinsonde, NCAR/ISS" after consultation with the scientific staff (Scot). The Station CD file was hand-edited to reflect a platform of "121."

    • $header->setType returns "GAUS SOUNDING DATA".

    • $header->setId returns "MISS".

  4. Verify that the raw data columns match the required ESC format (see example).

  5. Run the ISS_Converter script. This will convert the EOL data to the ESC format.

  6. Run the format checker using the ant command:

    ant check-format-ESC

    Check the log file in the output directory to determine if any format issues need to be addressed before proceeding with AutoQC.

Auto QC

This step works best on merlot, but can be done on any machine with the proper Ant and Java setup.

These steps execute the automatic QC on the processed files in the output directory, generate log files in the logs directory, and place the QC'ed data files into the final directory, as specified by the build.xml file. After the QC processing has completed, the ant autoqc command runs the format checker on the QC'ed files. Here are the steps to run the sounding Automatic Quality Control Processing:

  1. Stay in/Change to the top level processing directory where the Ant build.xml file is located.

  2. Check the build.xml file for the autoqc target for the following:

    1. A command to copy the properties file to the build directory. There is typically a copy of the autoqc properties file (e.g., "us_plains_autoqc.properties") in the software directory.

    2. The java command(s) will be executed on the expected data in the expected locations (i.e., ensure that the directory names in the build.xml file are correct and what you want.)

    3. You have updated the project name in the build.xml, if included in the build.xml file.

  3. Perform the auto QC by executing the ant command:

    ant autoqc

    For PLOWS 2008-2009:

    • Format checker warnings for differences between surface altitude and header altitude were seen. Header altitude used GPSalt while GeoPoAlt was used in the actual data.

Visual QC

Once the auto QC is completed, email the scientific staff (i.e., Scot Loehrer) so that they can perform a visual QC on the data and give final approval of the processed data set, as required. Note that the visual qc is not performed on all sounding datasets.

Finalization

After the visual QC is completed, the data set is ready to be finalized (i.e, prep'd for CODIAC).

  1. Update this "How To" document to indicate the latest project this conversion was executed for and include any notes about upgrades made specifically for that project, as required. A copy of this document is generally found in the /docs directory. Place a copy of the updated doc in /net/web/dmg/html/software/conversions/upper_air/MISS directory. The s/w repository page points to this doc in that web location, so this must be updated. The s/w repository page is located here.

  2. Update the dataset description (or EMDAC) document for this conversion. Work with the scientific staff to create an EMDAC document. Generally, start with an existing "similar" document and simply update it for this project and dataset. Generally, a copy of the EMDAC doc can be found in the /docs area.

  3. The next processing step is to create day files that can/will be loaded into the EMDAC/CODIAC database, along with the EMDAC dataset description document. The dayfiles creation step will generate day files sorted by nominal release date followed by station location as required by CODIAC. It's very important that the output dayfiles be in this expected sort order so that CODIAC can access all the soundings within a single dayfile. Do the following steps to create the day files.

    1. Ensure that you are in the same (top) directory as the build.xml file. Create the day files using the ant command:

      ant dayfiles

      This will create a file for each day where at least one sounding exists in the database directory or whichever directory is specified as the output directory in the build.xml file. If this dataset is to be put online "as is", these ar the dayfiles to load into CODIAC.

5mb Extraction

If the converted soundings are also going to be placed into a 5mb composite, the soundings need to have the proper 5mb pressure level soundings generated. This is done through the following steps:

  1. Generate the 5mb pressure level data files using the ant command:

    ant 5mb-extract

    This will generate a 5 mb sounding file for each sounding in the final directory and place it in the 5mb. It will also run some final checks to make sure the format is correct. Beware that the file directories listed in the build.xml file are what you expect and want.

    For PLOWS 2008-2009:

    • Because the build.xml file expected to see files with a .cls extension, the non-qc'd files in the final directory were moved out to the top level directory into a "NonQC" directory, and the files with the .qc extension were renamed.

The generated 5mb data files will then be available when the composite is generated.
