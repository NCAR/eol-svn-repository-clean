Purpose

This document contains instructions for converting Ishigaki Jima High Resolution Radiosonde (JMA - Japanese Meteorological Society) data into the EOL Sounding Composite (ESC) format.

NOTE: This converter may be hard to reuse for other data, as it contains many hard-coded fixes that pertain to Ishigaki Jima specifically.

Directory Structure

Ishigaki_Jima/

5mb/The directory where the extracted 5mb pressure sounding files are stored.
build/The auto-generated directory for the compiled Java classes.
dayfiles/The directory where the generated day files are stored. This is the final processed data that will be put online.
docs/The directory containing documentation about the processing and the data.
final/The directory where the QC'ed files are placed once they have been created.
logs/The directory where the error/check files from the QC are placed.
output/The directory where the ESC files are put after they are generated.
raw_data/The directory where the raw data is stored.
software/The directory where the software for the conversion is stored.
src/Unused directory - required for autoqc to run.

Processing the Data

Initial Processing Setup

  1. Copy the raw data into the raw_data directory. Check the assumptions made in the script regarding things such as file names and locations.

    For T-PARC:

    • There were 186 files in two directories. Some of the files started with a "-" which was removed and replaced with an "F" to match the other raw data files.

    • Ran dos2unix utility on the raw data files to remove control codes.

Converting the Data

This step runs Perl code and should work on basically any machine. However, to run the autoQC and any other processing/data prep done by the build.xml (Ant) file, you must run on any machine with the proper Ant and Java setup, such as merlot.

Converting Ishigaki Jima High Resolution Radiosonde (JMA) data into the ESC format consists of the following steps.

  1. Change to the software directory.

  2. Edit the Japanese_Radiosonde_Converter.pl script.

    • BEWARE: The converter expects the actual data to begin on line 8 of the raw data file, but the data records should be discarded until the counter (second column) resets to zero. That data record becomes the surface data record..

    • Search for "HARD-CODED" to find and change the hard-coded project-specific values in the script to be correct for your project. For example, change the function: getProjectName to the current project.

    • Code was added to the converter (from the Ron Brown Converter) to derive the ascent rate values from the altitude and time data.

    • For T-PARC Ishigaki Jima data:

      • One raw data file contained header start date/time information different from the date/time reflected in the file name and identical to a second raw data file. Scot reviewed the two files and determined that the one with the mismatched date information should be removed from the raw_data. This file was put in the unprocessed_raw_data directory until such time as we hear back from the data source.

      • The time for each data record in the raw data is in the HH:MM:SS format, starting with the release time for the surface record. Code in the converter sets the time to "0" seconds for the surface record, then converts the HH:MM:SS time for each record into seconds and compares it with the previous record. This was done because of the many gaps in this 1-second data. If there is a gap in time, a record with all missing values will be inserted into the output to keep the timing consecutive. This was a common occurrence throughout the raw data.

      • BEWARE - MANUAL CORRECTION TO RAW DATA FILE:   One raw data file had a data record with an incorrect time. This caused a gap of 13,185 "missing" records. The time was manually corrected at the direction of the science staff.

      • BEWARE - MANUAL CORRECTION TO RAW DATA FILE:  One raw data file contained unreadable characters in the middle of the file. This section was manually removed from the file in order to allow processing of the file.

      After the intial conversion was reviewed, Scot L. requested that additional changes be made for the Ishigaki Jima data:

      • Six files did not contain good data and were removed from the raw_data directory and placed in the unprocessed_raw_data directory. See the README.txt file in that directory for more information.

      • For time values t=1 through t=10, all values except time were set to missing. The radiosonde was being acclimatized during this period and the data were not usable.

      • For any negative altitude, all values except time were set to missing.

      • Added @record_list array: All data records were read in and then processed in reverse order to remove the descending data that was included at the end of the file (i.e., negative ascension rate).

      • Some files contained more than one counter reset to zero. Code was added to skip the first reset zero for three files, and skip the second reset zero for another file. The filenames were HARD-CODED in.

      After the final changes were made and the data was processed, it was noticed that the time in the raw data was local time. At Linda Cully's suggestion, a post-processing converter (ConvertToUTC.pl) was written to read in the processed files, convert local time to UTC time, write out the file to a new file with the UTC file name, and change the two affected header lines (UTC Release and Nominal Release). This script was run on all the files in both the output/ and the final directories.

  3. Run the Japanese_Radiosonde_Converter.pl script. This will convert data to the ESC format.

  4. Run the format checker using the ant command:

    ant check-format-ESC

  5. Check the log file (generated by the format checker) in the output directory to determine if any format issues need to be addressed before proceeding with AutoQC.

Auto QC

This step works best on merlot, but can be done on any machine with the proper Ant and Java setup.

These steps execute the automatic QC on the processed files in the output directory, generate log files in the logs directory, and place the QC'ed data files into the final directory, as specified by the build.xml file. The final part of the automated process is to run the format checker on the QC'ed files. To run the sounding Automatic Quality Control Processing:

  1. Stay in/Change to the top level processing directory where the Ant build.xml file is located.

  2. Check the build.xml file for the autoqc target for the following:

    1. A command to copy the properties file to the build directory. There is typically a copy of the autoqc properties file (e.g., "us_plains_autoqc.properties") in the software directory.

    2. The java command(s) will be executed on the expected data in the expected locations (i.e., ensure that the directory names in the build.xml file are correct and what you want.)

    3. You have updated the project name in the build.xml, if included in the build.xml file.

  3. Perform the auto QC by executing the ant command:

    ant autoqc

  4. After all issues have been resolved, notify the Scientific Staff (Scot) that the data is ready for Visual QC. Also inform him of any issues noted so that he will be aware of them.

Visual QC

Once the auto QC is completed, email the scientific staff (i.e., Scot Loehrer) so that they can perform a visual QC on the data and give final approval of the processed data set, as required. Note that the visual qc is not performed on all sounding datasets.

Finalization

After the visual QC is completed, the data set is ready to be finalized (i.e, prep'd for CODIAC).

  1. Update this "How To" document to indicate the latest project this conversion was executed for and include any notes about upgrades made specifically for that project, as required. A copy of this document is generally found in the /docs directory. Place a copy of the updated doc in /net/web/dmg/html/software/conversions/upper_air/Ishigaki_Jima directory. The s/w repository page points to this doc in that web location, so this must be updated. The s/w repository page is located here.

  2. Update the dataset description (or EMDAC) document for this conversion. Work with the scientific staff to create an EMDAC document. Generally, start with an existing "similar" document and simply update it for this project and dataset. Generally, a copy of the EMDAC doc can be found in the /docs area.

  3. The next processing step is to create day files that can/will be loaded into the EMDAC/CODIAC database, along with the EMDAC dataset description document. The dayfiles creation step will generate day files sorted by nominal release date followed by station location as required by CODIAC. It's very important that the output dayfiles be in this expected sort order so that CODIAC can access all the soundings within a single dayfile. Do the following steps to create the day files.

    1. Ensure that you are in the same (top) directory as the build.xml file. Create the day files using the ant command:

      ant dayfiles

      This will create a file for each day where at least one sounding exists in the database directory or whichever directory is specified as the output directory in the build.xml file. If this dataset is to be put online "as is", these ar the dayfiles to load into CODIAC.

5mb Extraction

If the converted soundings are also going to be placed into a 5mb composite, the soundings need to have the proper 5mb pressure level soundings generated. This is done through the following steps:

  1. Generate the 5mb pressure level data files using the ant command:

    ant 5mb-extract

    This will generate a 5 mb sounding file for each sounding in the final directory and place it in the 5mb. It will also run some final checks to make sure the format is correct. Beware that the file directories listed in the build.xml file are what you expect and want.

The generated 5mb data files will then be available when the composite is generated.
