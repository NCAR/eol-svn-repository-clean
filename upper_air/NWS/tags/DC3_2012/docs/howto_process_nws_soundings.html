Purpose

This document contains the process of how to convert the National Weather Service (NWS) upper air sounding data EOL receives from NCDC from their ASCII file formats to the ESC format.

If you make any changes to the software for either the main trunk, be aware that the same changes MAY be required for the NWS_BNL conversion s/w. This is because the NWS_BNL conversion is almost identical to the NWS general processing. Note that in the current repository under the NWS upper_air converter, there exists a BNL (Brookhaven National Laboratory) tag. DO NOT UPDATE that tag in any way. Leave that portion of the repository as is. As of August 2009, an /upper_air/NWS_BNL area was created in the repository. Form this point forward, the best and most complete version of the NWS_BNL conversion s/w can be found at this NWS_BNL location in the repository.

Directory Structure

NWS/

5mb/The directory where the extracted 5mb pressure sounding files are stored.
build/The auto-generated directory for the compiled Java classes for the RRS conversion.
database/The directory where the final processed data that will be put online is stored.
docs/The directory containing documentation about the processing and the data.
final/The directory where the auto QC'ed MicroART files are placed as they are created.
final_merged/The directory where the best auto QC'ed files for both MicroART and RRS are placed in preparation for the visual QC. The visual QC results are also placed in this directory.
lib/The directory containing libraries for the MicroArt conversion.
logs/The directory where the error/check files from the QC are placed.
matrix_lib/The directory that contains the source code and header files for the matrix library used by the NWS MicroArt conversion programs.
nws_src/The directory that contains the source code for the NWS conversion programs.
output/The directory where the processed MicroART files are put after they are formatted to ESC.
raw_data/The directory where the MicroART raw data is stored for the conversion.
rrs_final/The directory where the auto QC'ed RRS files are places as they are created.
rrs_output/The directory where the processed RRS fiels are put after they are formatted to ESC.
rrs_raw_dataThe directory where the source RRS files are stored for the conversion.
software/The directory where the executable software for the MicroArt conversion is stored.
src/The directory where the Java source files for the RRS conversion are stored.
vasil_MT2_converter/The directory where the Vasil MT2 -> txt sounding converter for special NWS data is stored. This must be downloaded and installed on a Windows machine.

Other Directories

/net/work/operational/sounding/NWS/archive/ascii_filesThe location where the MicroArt ASCII data files are stored after being inventoried. These are the files (i.e., raw MicroArt data) to convert.
/net/work/operational/sounding/NWS/archive/rrs_textThe location where the RRS ASCII data files are stored after being extracted from the buffer files and archived. These are the files (i.e., raw RRS data) to convert.

Processing the Data

Initial Processing Setup

There are four things that must be done before processing the data.

  1. Creating the Matrix Library: This is special library downloaded from the internet that is used by the MicroArt converter. Instructions on creation can be found here.
  2. Build the Executables: The MicroART conversion software is in C/C++ and needs to be compiled to an executable to be run. The instructions on doing this can be found here.
  3. Setup Ant: The RRS conversion uses an Ant build configuration and must be setup in your environment. If your environment has already been setup for Ant, this step may be skipped. To setup your environment for Ant, see: Ant Setup and Use.
  4. Copy the source data file to the appropriate directories. The MicroART files go in raw_data and the RRS files go in rrs_raw_data.

Convert the MicroART Data

This must be done on a Solaris OS, since the matrix library and executables will only be created on that OS.

This step will convert the MicroART ASCII files into ESC ASCII files. If you don't have MicroArt data to convert, you can skip this set of steps and go directly to "Convert the RRS Data".

  1. Change to the software directory.
  2. Make sure both the nwsVaisala and nwsVIZ executables are in the software directory. If they are not, see Build the Executables.
  3. Update the getProjectName() function to return the correct project ID in the NWS_Converter.pl script.
  4. Run the NWS_Converter.pl script. This will convert all of the files in the raw_data to ESC sounding files and generate the appropriate station list.

This can be a slow process at about 20-30 minutes per month file.

If you are processing BNL data, YOU SHOULD FOLLOW THE NWS BNL CONVERSION DOCUMENT WHICH WILL INSTRUCT YOU TO continue with the following auto QC step. Any other conversion should use the auto QC described here after converting the RRS data.

  1. Always run the autoqc software via the build.xml file. Inspect that file for the proper call to make which is generally, "ant autoqc". Ensure that the build.xml file will QC the proper data. Running the autoqc will QC each ESC file in the output and put the resulting file in the final directory and a log file in the log directory.

BEWARE: Running the NWS_Converter.pl (instead of the BNL version) on the BNL data will result in output file names with the actual times in the file names instead of the expected nominal times required for BNL!!! Don't do it!

Convert the RRS Data

This step works best on merlot, but can be done on any machine with the proper Ant and Java setup.

  1. Ensure that you have copied all the raw data files into the /rrs_raw_data directory. These raw files can be found in the year and station directories under the /net/work/operational/sounding/NWS/archive/rrs_text/ area. Copy all the *Meta.txt, *PTU.txt, and *GPS.txt files in the /rrs_raw_data directory.

  2. Change to the top level processing directory of the conversion. (Where the build.xml file is located.)

  3. This step will convert the RRS binary BUFR files or files that were previously extracted txt files into ESC ASCII files.

    This step is only required if the ASCII RRS files are unavailable and have to be generated from the buffer files. Extract the RRS BUFR files into txt files: This uses the RRS Buffer Extraction Software to extract the ASCII text files from the BUFR files.

    Execute the Ant command:

    ant unbufr

  4. Update the PROJECT_ID constant in the src/dmg/ua/sounding/convert/nws/RRSSoundingConverter.java file to contain the correct project ID.

  5. Convert the RRS ASCII text files into ESC files. This ant convert command will read and combine the information from the meta data, PTU, and wind files (i.e., *_1Meta.txt, *_5pPTU.txt, *_6pGPS.txt) for each sounding into a single output ESC sounding file. The output *.cls files will be placed in the rrs_output directory. This ant convert command will also run the format checker on the output data, sort the records (and rename the original output files to .unsort), and then run the format checker on the sorted record files.

    Execute the Ant command:

    ant convert

    to convert the *.txt input files to *.cls output files.

Auto QC

This step works best on merlot, but can be done on any machine with the proper Ant and Java setup.

This step executes the automatic QC on the processed files (/output and /rrs_output), generates log files (/logs), and places the QC'ed files in the final directory specified by the build.xml file (/final &/or /rrs_final). After autoQC runs, the build.xml file will run the format checker on the QC'ed files.

BEWARE: that the build.xml script runs the "NWSESCAutoQC" version of the autoQC s/w on the MicroArt data BUT build.xml runs the more general "UpsondeESCAutoQC" version of the autoQC s/w on the RRS data. The MicroArt form of the data is being phased out and has very specific issues that require a certain form of the autoQC to run on it. The RRS form of the NWS data is the data format being transitioned to. This RRS form no longer has the issues that the MicroArt data does/did, so the RRS data only requires the more typical form of the upsonde autoQC.

  1. Stay in/Change to the top level processin directory where the Ant build.xml file is located.
  2. Check the build.xml file for the autoqc target and make sure there are:
    1. Command to copy the property files to the build directory.

      Beware that there are different versions of the properties files required to run autoQC. There are "HOT" and "COLD" properties files and there are upsonde, dropsonde, and driftsonde properties files. For the NWS data which is all in the United States, we use the us_plains_autoqc.properties files (typically found in the /software dir). The us_plains_autoqc.properties files is used for "HOT" upsondes.

    2. See the build.xml file to determine the exact location and form of the input data QC'd and the output data generated. Ensure that these are correct.

  3. Perform the auto QC by executing the ant command:

    ant autoqc

  4. Merge the MicroART and RRS final data files together into a single directory to prepare for the visual QC. If you only have one type of data, you don't need to do this step. See the build.xml file for the exact files combined and the directory the files are placed. This is done by the command:

    ant merge

Visual QC

Once the auto QC is completed, the scientific staff (generally Scot Loehrer) will perform a visual QC on the data and do final approval of the processed data set.

Scot uses a piece of s/w that will only execute on a Solaris OS like Cabernet (or Pandora). Believe that s/w tool which generates skew-T plots will only run on Cabernet and probably was rebuilt on an older version of the OS. Believe Mark Bradford originally wrote this s/w.

Visual QC generates *.cls.qc files generally in /rrs_final for the RRS data and in /final for the MicroArt data.

Final Preparation

After the visual QC is complete, the data set is ready to be prepared for database entry. This entails creating day files and possibly loading the data set into the archive database.

For General Projects

The following ant dayfiles command will generate day files sorted by nominal release date and then by station location as required by CODIAC.

  1. Create the day files using the ant command:

    ant dayfiles

    This creates an output *.cls file in the database directory for each day where there exists at least one sounding in the final_merged directory or whichever input directory is specified in the build.xml file.

5mb Extraction

If the converted soundings are also going to be used in a 5mb composite, the soundings need to have 5mb pressure level soundings generated. This is done by doing the following:

  1. Generate the 5mb pressure level data files using the ant command:

    ant 5mb-extract

    This will generate a 5 mb sounding file for each sounding in the final_merged directory and place the output 5mb file in the 5mb directory. It will also run final checks to make sure the format is correct. See the build.xml file for the exact command executed. These 5mb files are the ones to use in the 5mb composite dataset.

Building Libraries and Executables

Creating the Matrix Library

The following currently only works on a Solaris OS and works best on tornado.

The matrix library is a download from the internet that is dated from 1992-1995. As of April 2005, the online version is the same as the EOL version and has not been updated. The EOL version has been updated to work with the current version of g++, which the online version does not. The change was to add the const keyword to the matrix::matrix(matrix &y) function so now the function prototype looks like: matrix::matrix(const matrix &y).

The following steps are how to generate the matrix library, libmat.a:

  1. Change to the matrix_lib directory. This contains all of the header and code files for the library, along with the makefile: libmat.mak to generate the library.
  2. Create the library:

    make -f libmat.mak

    This will create the library matrix_lib.a in the lib directory and copy the needed matrix.hpp file to the lib directory as well. To make the library issue the command listed directly above.

  3. Change to the lib directory to make sure that the libmat.a library and the matrix.hpp header file both are in the directory.

Creating the Conversion Programs

The following currently only works on a Solaris OS.

The following steps will create the binaries for converting the MicroART data into the ESC format.

  1. Change to the nws_src directory.
  2. Run the command to make the binaries and put them into the software directory.

    make -f MakeNWS

Vasil File Converter

The Vasil File Converter is a Windows program that Scot was given on a CD and was placed within the repository. It is used to convert certain NWS soundings from a binary format into ASCII files. The soundings are for a small number of stations which are not part of the continual archive sent by NCDC. The stations include Charleston, SC; Albany, NY; and a couple of others.

The Vasil program generates ASCII text files which are then converted to ESC with the NWS_Converter.pl script.

Converting to EBUFR

The following explains how to EBUFR the QC'ed CLASS files. This assumes that the class2ebufr binary file has been generated. For more information on the EBUFR process, see How To: EBUFR CLASS Sounding Data.

  1. Change to the software directory (if you are not already there).
  2. Make sure the EBUFR configuration files are in the directory. There should be five (class_file, code_08_021, code_33_254, control_scf.txt, desc_file). The program will crash without these files.
  3. Make sure there is a copy of the executable (or a symbolic link to it) in the directory.
  4. Edit the control_scf.txt file.

    On the 12th line, (two lines above the "END FILE" line) the project name needs to be edited to match the current project. The rest of the files should remain untouched.

  5. Edit the create_ebufr_files.pl script.
    1. Change the variable: $PROJECT to the name of the current project.
    2. Change the variable: $PROGRAM to the path of where the class2ebufr program for the current project is located.
  6. Run the create_ebufr_files.pl script. This will concatonate the CLASS files into day CLASS file which will be used to generate the EBUFR day files.

    Near the end of the file is a pattern for generating the names of the outputed ebufr files. Make sure the network name is changed to NWS.

    The script copies the five aforementioned files to the ebufr directory and removes them once the processing is finished. This is to force all of the intermediate and final files to be saved in the ebufr directory.
